Снижение количества ложных срабатываний

Данный проект реализует многоэтапный анализ уязвимостей зависимостей Python-проекта с целью снижения количества ложных срабатываний. В отличие от классических SCA-инструментов, которые определяют уязвимости только по версии библиотеки, в данном решении учитывается контекст использования кода, анализируются патчи уязвимостей и проводится дополнительная проверка во время выполнения программы.

В качестве входных данных используется отчет Trivy, сформированный на основе файла зависимостей requirements.txt. Пример зависимостей включает django версии 2.2, flask версии 1.0 и requests версии 2.19.1. Работа системы представляет собой последовательный pipeline, где каждый шаг обрабатывает результат предыдущего и добавляет новый уровень анализа.

1. Парсинг отчета Trivy

На первом этапе запускается скрипт parse_trivy.py.

Скрипт читает файл report.json, сформированный Trivy, извлекает из него список уязвимостей, включая идентификатор CVE, имя пакета, установленную версию, исправленную версию, уровень критичности и заголовок. Далее происходит удаление дубликатов и сортировка уязвимостей по уровню критичности.

Результатом выполнения является формирование файлов cves.json и cves.txt в каталоге data/processed. Файл cves.json используется для дальнейшей автоматической обработки, а cves.txt предназначен для просмотра человеком.

2. Фильтрация по версиям

Далее запускается скрипт filter_real_vulns.py.
На этом этапе производится сравнение установленной версии библиотеки с версиями, в которых уязвимость исправлена. Если установленная версия ниже хотя бы одной исправленной версии, уязвимость считается потенциально применимой.
Результат сохраняется в файл real_vulns.json. На данном этапе все уязвимости, обнаруженные Trivy, обычно остаются, так как используются устаревшие версии библиотек.

3. Получение данных из NVD

Следующим шагом является запуск скрипта nvd_lookup.py.
Скрипт выполняет запросы к базе данных NVD и получает дополнительные сведения по каждой CVE, в частности список ссылок на источники, включая GitHub, advisory-страницы и другие ресурсы. Для оптимизации используется механизм кэширования, при котором результаты сохраняются в файл nvd_cache.json, что позволяет избежать повторных запросов к API.
Результаты обработки сохраняются в файл patch_links.json.

4. Формирование контекста CVE

Скрипт build_cve_context.py объединяет информацию из cves.json и nvd_cache.json, формируя единый контекст для каждой уязвимости. В контекст включаются идентификатор CVE, пакет, версия, уровень критичности, заголовок и ссылки на источники.
Результат сохраняется в файл cve_context.json.

5. Классификация уязвимостей

Далее выполняется скрипт classify_cves.py.
На этом этапе уязвимости классифицируются по категориям на основе анализа текста заголовка и описания. Используются такие категории, как аутентификация, шаблоны, SQL-инъекции, загрузка файлов, сессии и другие. Классификация осуществляется на основе ключевых слов.
Результат сохраняется в файл cve_classified.json.

6. Анализ использования кода проекта

Скрипт scan_project_usage.py выполняет анализ исходного кода проекта, расположенного в каталоге demo_project.
С помощью регулярных выражений определяется, какие механизмы действительно используются в коде, например аутентификация, работа с шаблонами или сессиями.
Результат сохраняется в файл project_usage.json.

7. Сопоставление уязвимостей с использованием

Скрипт match_cves_to_usage.py сопоставляет категории уязвимостей с фактически используемыми механизмами в проекте.
Если категория уязвимости совпадает с используемой категорией, она помечается как likely_relevant, в противном случае как likely_not_relevant.
Результат сохраняется в файл cve_relevance.json.

8. Формирование отчета по релевантным уязвимостям

Скрипт make_relevant_report.py формирует итоговый список уязвимостей, которые были признаны релевантными.
Результат сохраняется в файл relevant_report.json и дополнительный текстовый отчет.

9. Поиск патчей
Скрипт extract_patch_from_advisory.py анализирует ссылки из patch_links.json, загружает advisory-страницы и извлекает из них ссылки на патчи, коммиты и pull request.
Результат сохраняется в файл patch_candidates.json.

10. Анализ патчей

Скрипт analyze_patch_links.py загружает найденные патчи и извлекает из них информацию об измененных файлах и фрагментах кода.
Результат сохраняется в файл patch_analysis.json.

11. Сопоставление патчей с кодом проекта

Скрипт match_patch_to_project.py проверяет, затрагивают ли изменения из патчей файлы проекта.
Если измененные файлы совпадают с файлами проекта, уязвимость помечается как possibly_exploitable, иначе как likely_not_exploitable.
Результат сохраняется в файл final_analysis.json.

12. Runtime-анализ
Скрипт run_demo_with_monitor.py запускает код проекта с использованием runtime_monitor.py, который отслеживает вызовы функций во время выполнения.
Результат работы сохраняется в файл runtime_log.txt.

13. Финальное обогащение результатов

Скрипт enrich_with_runtime.py объединяет результаты статического анализа с runtime-данными.
Если категория уязвимости подтверждается во время выполнения, уровень уверенности повышается.
Итоговый результат сохраняется в файл final_with_runtime.json.
